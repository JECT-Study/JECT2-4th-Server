## 1. 신규 도메인

> **Vote 도메인은 본 명세 범위 외.** 알림 도메인은 `vote_id` FK만 참조하며,
> 알림 발송 시점의 `vote.title` / `vote.thumbnail_url`은 알림 row에 **스냅샷으로 저장**한다.
> 이렇게 두는 이유:
> 1. 알림이 후행 vote 변경 / 삭제로부터 독립
> 2. 목록 조회 시 vote 테이블과 join 불필요 → 인덱스만으로 응답

### Notification (알림)

```sql
CREATE TABLE notification (
    id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id       BIGINT NOT NULL REFERENCES users(id),
    type          VARCHAR(40) NOT NULL,                              -- VOTE_RESULT_PUBLISHED 등
    vote_id       BIGINT REFERENCES vote(id),                        -- 클릭 시 이동 대상 (nullable)
    title         VARCHAR(200) NOT NULL,                             -- 발송 시점 vote.title 스냅샷
    body          VARCHAR(500) NOT NULL,                             -- "투표 결과가 공개됐어요" 등
    thumbnail_url VARCHAR(512),                                      -- 발송 시점 vote.thumbnail_url 스냅샷
    is_read       BOOLEAN NOT NULL DEFAULT FALSE,
    read_at       TIMESTAMP WITH TIME ZONE,
    created_at    TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);

CREATE INDEX idx_notification_user_created ON notification (user_id, created_at DESC);
CREATE INDEX idx_notification_user_unread  ON notification (user_id) WHERE is_read = false;
```

### NotificationSetting (알림 설정)
사용자별 1:1. 푸시 ON/OFF 상태 + 알림 설정 유도 팝업(1-8) 노출 카운트 관리.

```sql
CREATE TABLE notification_setting (
    user_id              BIGINT PRIMARY KEY REFERENCES users(id),
    push_enabled         BOOLEAN NOT NULL DEFAULT FALSE,
    push_enabled_at      TIMESTAMP WITH TIME ZONE,
    push_disabled_at     TIMESTAMP WITH TIME ZONE,
    prompt_dismiss_count INT NOT NULL DEFAULT 0,                     -- "나중에 할게요" 누적 횟수
    last_prompt_shown_at TIMESTAMP WITH TIME ZONE,                   -- 마지막 1-8 팝업 노출 시각
    created_at           TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
    updated_at           TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now()
);
```

> 회원가입 시 row 자동 생성 (default `push_enabled = false`). User 도메인 담당자와 협의 — 회원가입 트랜잭션에서 함께 INSERT하거나, 본 도메인의 `@EventListener(UserCreatedEvent.class)`로 처리.

### PushSubscription (Web Push 구독 정보)
브라우저(Chrome / Safari)가 발급한 Push endpoint와 클라이언트 공개키. 동일 사용자가 여러 기기에서 구독 가능 (PC + 모바일).

```sql
CREATE TABLE push_subscription (
    id           BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id      BIGINT NOT NULL REFERENCES users(id),
    endpoint     VARCHAR(2048) NOT NULL UNIQUE,                      -- FCM / APNs / Mozilla autopush URL
    p256dh_key   VARCHAR(255) NOT NULL,                              -- ECDH 클라이언트 공개키 (base64url)
    auth_key     VARCHAR(255) NOT NULL,                              -- 인증 시크릿 (base64url)
    user_agent   VARCHAR(512),                                       -- 디버깅용 (어떤 기기인지)
    created_at   TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
    last_used_at TIMESTAMP WITH TIME ZONE
);

CREATE INDEX idx_push_subscription_user ON push_subscription (user_id);
```

> `endpoint`는 푸시 서비스(FCM 등)가 발급한 URL. 사용자가 구독 해제하거나 브라우저가 만료시키면 발송 시 `410 Gone`이 떨어지는데, 이 경우 해당 row를 삭제한다. (자세한 처리는 plan 문서)

## 2. REST API

> 모든 엔드포인트 공통 Header: `Authorization: Bearer {accessToken}`
> 모든 엔드포인트는 **회원 전용**. 비회원이 호출하면 `401`.

---

### GET `/api/notifications` — 알림 목록

알림 페이지 진입 시 호출. 최대 1개월 이내 알림만 반환.

**Request**
| 구분 | 파라미터 | 타입 | 필수 | 설명 |
|------|----------|------|------|------|
| Query | `cursor` | `Long` | 선택 | 이전 페이지 마지막 notificationId. 없으면 최신부터 |
| Query | `size`   | `Int`  | 선택 | 페이지 크기 (기본값: 20) |

**Response**
```json
{
  "notifications": [
    {
      "notificationId": 1042,
      "type": "VOTE_RESULT_PUBLISHED",
      "voteId": 88,
      "title": "주 4일 근무 도입, 찬성 vs 반대",
      "body": "투표 결과가 공개됐어요",
      "thumbnailUrl": "https://cdn.example.com/votes/88/thumb.jpg",
      "isRead": false,
      "createdAt": "2026-04-28T12:23:00+09:00"
    }
  ],
  "nextCursor": 1020,
  "hasNext": true
}
```

정렬: `createdAt DESC`
필터: `created_at >= now() - INTERVAL '1 month'` 만 반환 (디자인 명세상 "최대 1개월 전까지")

> 시간 표기 (`2시간 전` / `3일 전` / `4월 3일`)는 프론트에서 `createdAt` 기준으로 계산. 백엔드는 ISO 8601 그대로 내려준다.

---

### GET `/api/notifications/unread-count` — 안 읽은 알림 개수

알림 화면 상단(`1-2 읽지 않은 알림 N개`) 및 홈 메인의 알림 버튼 뱃지에 사용.

**Response**
```json
{ "unreadCount": 3 }
```

> 99 초과여도 정수 그대로 반환 (`{ "unreadCount": 152 }`). `99+` 표시는 프론트가 처리.

---

### POST `/api/notifications/{notificationId}/read` — 개별 읽음 처리

알림 카드 클릭 시 호출 (이동 + 읽음 처리). 본인의 알림이 아닐 경우 `403`.

**Request**
| 구분 | 파라미터 | 타입 | 필수 | 설명 |
|------|----------|------|------|------|
| Path | `notificationId` | `Long` | 필수 | 알림 ID |

**Response:** `204 No Content`

> 멱등 처리. 이미 읽음 상태여도 200 계열 응답으로 처리하고 `read_at`은 갱신하지 않는다.

---

### POST `/api/notifications/read-all` — 모두 읽음 처리

`1-3 / 1-4 모두 읽음` 버튼 클릭 시 호출. 본인의 모든 미읽음 알림을 일괄 read 처리.

**Response:** `204 No Content`

```sql
UPDATE notification
   SET is_read = true, read_at = now()
 WHERE user_id = :userId AND is_read = false;
```

---

### GET `/api/me/notification-setting` — 푸시 알림 설정 조회

마이페이지 토글(`2-3 OFF / 2-4 ON`) 초기 상태 표시용.

**Response**
```json
{
  "pushEnabled": true,
  "pushEnabledAt": "2026-04-28T10:00:00+09:00"
}
```

> `pushEnabled = false` 일 땐 `pushEnabledAt`은 `null`. 거부 직전 토스트의 `"YYYY.MM.DD는 현재 거부한 날짜 표시"`는 별도 응답 필드를 두지 않고 클라이언트가 `pushDisabledAt`을 받아 처리할 수도 있는데, 디자인 명세에서 마지막 거부일 표시가 명확히 요구되면 응답에 추가:

```json
{
  "pushEnabled": false,
  "pushEnabledAt": null,
  "pushDisabledAt": "2026-04-15T18:30:00+09:00"
}
```

---

### PUT `/api/me/notification-setting` — 푸시 알림 ON/OFF

토글 변경 시 호출. **이 API는 DB 상태만 갱신**한다. 실제 OS 권한 요청 / PushSubscription 등록은 별도 API로 분리 (아래 두 단계).

**Request**
| 구분 | 파라미터 | 타입 | 필수 | 설명 |
|------|----------|------|------|------|
| Body | `pushEnabled` | `Boolean` | 필수 | `true` ON / `false` OFF |

```json
{ "pushEnabled": true }
```

**Response**
```json
{
  "pushEnabled": true,
  "pushEnabledAt": "2026-04-28T14:24:00+09:00"
}
```

> 토글 ON 시 흐름:
> 1. (프론트) PWA 환경 체크 → A2HS 미설치면 설치 유도 모달 노출
> 2. (프론트) PWA 설치 + 앱 실행 + OS 권한 요청
> 3. (프론트) `navigator.serviceWorker.ready.pushManager.subscribe(...)` → PushSubscription 객체 획득
> 4. (REST) `POST /api/me/push-subscription` ← PushSubscription 등록
> 5. (REST) `PUT /api/me/notification-setting` ← `pushEnabled = true` 갱신
>
> 4 → 5 순서를 지켜야 한다. 토글 ON인데 구독 row가 없으면 발송 단계에서 누락된다.

---

### POST `/api/me/push-subscription` — Web Push 구독 등록

브라우저가 발급한 PushSubscription 정보를 서버에 저장. 사용자가 여러 기기에서 구독 시 `endpoint UNIQUE` 제약으로 중복 방지 (동일 endpoint가 다시 들어오면 `last_used_at`만 갱신, upsert).

**Request**
```json
{
  "endpoint": "https://fcm.googleapis.com/fcm/send/eXaM...pLeEnDpOiNt",
  "p256dhKey": "BNc...4=",
  "authKey": "QGV...wA="
}
```

**Response**
```json
{ "subscriptionId": 7 }
```

> `endpoint` 값은 푸시 서비스마다 도메인이 다르다 (`fcm.googleapis.com`, `web.push.apple.com`, `updates.push.services.mozilla.com`). 백엔드는 도메인 검증을 하지 않고 받은 그대로 저장한다. 

---

### DELETE `/api/me/push-subscription` — 구독 해제

토글 OFF 시 호출. 본인의 모든 PushSubscription row를 삭제.

> 단일 endpoint만 해제하고 싶으면 `DELETE /api/me/push-subscription?endpoint=...` 형태로 query 받아 분기. 본 명세에서는 일괄 삭제만 지원 (디자인 명세상 토글이 단일이므로).

**Response:** `204 No Content`

---

### GET `/api/me/notification-prompt-status` — 알림 설정 유도 팝업(1-8) 노출 여부

투표 참여(`POST /api/votes/{voteId}/participate`) 응답 후 프론트가 호출. 응답이 `shouldShow: true`면 1-8 팝업("투표 결과를 알려드릴까요?") 노출.

**Response**
```json
{
  "shouldShow": true,
  "totalParticipationCount": 1
}
```

판정 로직:
1. `notification_setting.push_enabled = true` → `shouldShow = false` (이미 켜져있음)
2. 사용자의 누적 vote_participation 카운트가 `1`이거나 `(count - 1) % 10 == 0` → `shouldShow = true`
3. 그 외 → `shouldShow = false`

> 디자인 명세: "첫 투표 직후 1-8 팝업 노출. 이후 투표를 10회 진행할 때마다 1-4 팝업 반복 노출."
> 즉 1번째, 11번째, 21번째 참여 직후 노출.

---

### POST `/api/me/notification-prompt-dismissed` — 팝업 dismiss 기록

1-8 팝업에서 "나중에 할게요" 클릭 시 호출. `prompt_dismiss_count++`, `last_prompt_shown_at = now()`.

**Response:** `204 No Content`

> "설정하러 가기" 클릭 시는 클라이언트가 마이페이지로 이동하면 끝. 별도 호출 불필요.

## 3. 이벤트 기반 알림 발송

### 트리거: VoteEndedEvent

Vote 도메인의 `VoteCloseScheduler`가 endAt 지난 vote를 close할 때마다 `VoteEndedEvent` 발행 (`vote-implementation-tasks.md` STEP 6 참조).

```java
// vote/event/VoteEndedEvent.java (vote 도메인 측)
public record VoteEndedEvent(Long voteId) { }
```

### 핸들러: VoteEndedNotificationHandler

알림 도메인 측에서 `@EventListener` + `@Async` 로 listen. 두 가지 작업 수행:

1. **알림 row 생성**: 해당 vote에 참여한 회원(= `vote_participation.user_id IS NOT NULL`) 모두에게 `VOTE_RESULT_PUBLISHED` 알림 row를 batch insert
2. **Web Push 발송**: `push_enabled = true` 이면서 `PushSubscription`이 등록된 사용자에게 비동기 발송

```
VoteCloseScheduler.closeExpiredVotes()
        │ 매분 실행
        ▼
  for each expired vote:
        ├─ vote.markEnded() + save
        ├─ ApplicationEventPublisher.publishEvent(new VoteEndedEvent(voteId))
        │
        ▼ (비동기 별도 스레드)
  VoteEndedNotificationHandler.on(VoteEndedEvent)
        ├─ 1. 참여 회원 조회 (VoteParticipationRepository.findUserIdsByVoteId)
        ├─ 2. Notification 일괄 INSERT (saveAll)
        └─ 3. PushSenderPort.sendBatch(userIds, payload)
              └─ for each push_enabled user:
                    └─ PushSubscription 조회 → Web Push 발송
                          ├─ 200/201: last_used_at 갱신
                          ├─ 404/410: 해당 row 삭제 (gone)
                          └─ 5xx / timeout: 로그만 남기고 무시
```

### Web Push 발송 페이로드 (브라우저 Service Worker 수신)

```json
{
  "type": "VOTE_RESULT_PUBLISHED",
  "title": "투표 결과가 공개됐어요",
  "body": "주 4일 근무 도입, 찬성 vs 반대",
  "icon": "/icons/notification-192.png",
  "badge": "/icons/badge-72.png",
  "data": {
    "voteId": 88,
    "url": "/votes/88/result"
  }
}
```

> 페이로드 포맷은 프론트(Service Worker)와 합의한 후 확정. 위 구조는 일반적인 Web Push 페이로드 컨벤션을 따른 예시.

## 4. 권한 정책

| 상황 | 처리 |
|------|------|
| 비회원 → 알림 페이지 진입 | 프론트가 회원 안내 팝업("알림은 회원 전용 기능이에요") 노출. API 자체는 호출하지 않음 |
| 비회원 → 알림 관련 모든 API 호출 | `401 Unauthorized` |
| 회원 → 본인이 아닌 알림 read 처리 | `403 Forbidden` |
| 미지원 브라우저 (카카오톡 / 인스타그램 인앱 등) → PushSubscription 등록 시도 | (프론트) A2HS 미지원 안내 토스트 노출. 백엔드는 호출 자체가 들어오지 않음 |
| iOS 16.4 미만 → 알림 토글 ON 시도 | (프론트) "iOS 업데이트 안내" 모달 노출. 백엔드 호출 없음 |
| 알림 권한 영구 거부 (브라우저 OS 레벨) | (프론트) "휴대폰 설정에서 알림 권한을 허용해주세요" 토스트. 백엔드 호출 없음 |

> iOS 16.4 미만, 미지원 브라우저, OS 권한 영구 거부 케이스는 **모두 프론트엔드 영역**. 백엔드는 PushSubscription 등록 API가 호출되었을 때만 처리하면 된다.

## 5. 외부 의존성: VAPID & Web Push

Web Push 프로토콜(RFC 8030 + RFC 8292) 발송을 위해 VAPID(Voluntary Application Server Identification) 키 쌍이 필요.

| 항목 | 값 |
|------|----|
| `vapid.public-key` | `application.yml` (또는 시크릿) — 프론트에도 동일한 공개키 주입 (Service Worker가 `pushManager.subscribe({applicationServerKey})` 호출 시 사용) |
| `vapid.private-key` | `application.yml` 또는 환경 변수 — 백엔드만 보유 |
| `vapid.subject` | `mailto:dev@vsapp.example.com` 등 RFC 8292 권고 |

> 키 발급은 Java 코드 또는 `web-push` CLI(`npm install -g web-push; web-push generate-vapid-keys`)로 1회 수행 후 시크릿 매니저 / `application-prod.yml`에 박아둠.

라이브러리 후보 (plan 문서에서 확정):
1. `nl.martijndwars:web-push` (가장 사용량 많음, BouncyCastle 의존)
2. `com.googlecode.json-simple` + 직접 RFC 8030 구현 (의존성 최소화, 작업량 ↑)

## 6. 구현 우선순위

| 단계 | 내용 |
|------|------|
| 1단계 | DDL (V4) + NotificationSetting / Notification / PushSubscription 도메인 + Repository |
| 2단계 | 알림 조회 API (`GET /api/notifications`, `unread-count`) |
| 3단계 | 알림 읽음 처리 API (`read`, `read-all`) |
| 4단계 | NotificationSetting / PushSubscription API + 1-8 팝업 status / dismiss API |
| 5단계 | Web Push 클라이언트 (VAPID 설정 + `PushSenderPort` 구현) |
| 6단계 | `VoteEndedNotificationHandler` 비동기 이벤트 핸들러 |

## 7. 본 명세 범위 외

| 항목 | 사유 |
|------|------|
| iOS 16.4 미만 안내 / 미지원 브라우저 (인앱) 안내 | 프론트 영역 |
| A2HS (Add to Home Screen) 설치 유도 모달 / Step 1~7 플로우 | 프론트 + Service Worker 영역 |
| OS 권한 팝업 (시스템 다이얼로그) | 브라우저 / OS 영역 |
| 사용자 알림 카테고리 분리 (광고성 / 운영성 등) | 추후 알림 종류 추가 시 확장. 현재는 `VOTE_RESULT_PUBLISHED` 단일 |
| FCM 직접 사용 | Web Push 표준(RFC 8030)으로 통일. 안드로이드도 Chrome PWA가 내부적으로 FCM을 거쳐 동작하므로 별도 SDK 불필요 |
| 알림 클릭 시 딥링크 처리 | Service Worker의 `notificationclick` 이벤트에서 `clients.openWindow(data.url)` 호출. 프론트 영역 |
